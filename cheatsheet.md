## PNU Slate

### Public AI Brochure Infrastructure — Architecture Notes & Project Overview

> “연구원님들이 GitHub를 배우는 대신, GitHub가 연구원님들에게 맞추도록 만들었습니다.”

---

## 0. 문서 목적

이 문서는 단순한 README나 기술 문서가 아닙니다.

PNU Slate 프로젝트를:

- 왜 만들게 되었는지
- 어떤 문제를 해결하기 위해 설계했는지
- 왜 이러한 기술 선택을 하게 되었는지
- 공공기관 환경에서 왜 이 구조가 적합한지
- 어떻게 협업과 유지보수를 고려했는지

까지 포함하여 정리한 프로젝트 아키텍처 및 설계 기록 문서입니다.

또한:

- 발표 자료
- 프로젝트 소개
- 내부 공유 문서
- 유지보수 참고 자료
- 협업 가이드

로도 활용할 수 있도록 구성하였습니다.

---

## 1. 프로젝트 시작 배경

### 실장님 요구사항

프로젝트의 출발점은 다음과 같은 요구사항이었습니다.

> “브로슈어를 GitHub 기반으로 협업할 수 있게 만들자.”

하지만 실제 현장에서는 다른 문제가 존재했습니다.

---

### 현실적인 문제

연구원 및 실무 사용자 입장에서:

- GitHub 사용 경험이 부족하고
- Git 명령어 사용이 어렵고
- Markdown 및 CLI 환경이 익숙하지 않으며
- 충돌 해결이나 브랜치 관리가 사실상 어려운 상황이었습니다.

즉:

GitHub는 매우 강력한 플랫폼이지만,
실제 사용자들은 GitHub 중심의 워크플로우에 익숙하지 않았습니다.

---

## 2. 프로젝트 핵심 아이디어

프로젝트를 설계하면서 가장 중요하게 생각했던 방향은 다음과 같습니다.

> 사용자가 GitHub를 배우는 것이 아니라,
> GitHub가 사용자를 대신하도록 만드는 것.

즉:

- GitHub는 시스템 내부에서 자동화 엔진 역할만 수행하고
- 사용자는 CMS 인터페이스만 사용하며
- 복잡한 Git 작업은 모두 시스템이 처리하도록 설계했습니다.

이 철학이 PNU Slate 전체 구조의 핵심입니다.

---

## 3. 핵심 설계 철학

### 3-1. Zero-Cost Infrastructure

프로젝트의 가장 중요한 목표 중 하나는:

> “운영 비용이 거의 발생하지 않는 구조”

를 만드는 것이었습니다.

---

### 왜 비용 최소화가 중요했는가?

공공기관 환경에서는:

- 월 서버 비용 처리
- 유지보수 계약
- 담당자 변경
- 예산 승인
- 장기 운영 안정성

등이 매우 중요합니다.

특히:
“몇 년이 지나도 안정적으로 운영 가능한가?”
가 핵심 고려사항이었습니다.

그래서 초기 설계 단계부터:

- 서버 제거
- 데이터베이스 제거
- 별도 백엔드 제거

를 목표로 삼았습니다.

---

## 4. 전체 시스템 구조

최종적으로 프로젝트는 완전한 Serverless Static Architecture 형태로 구성되었습니다.

```txt
관리자
 ↓
CMS
 ↓
GitHub REST API
 ↓
Repository Commit
 ↓
GitHub Actions
 ↓
Static Build
 ↓
GitHub Pages
```

즉:

- GitHub가 저장소 역할
- GitHub가 배포 서버 역할
- GitHub가 CI/CD 역할
- GitHub가 협업 플랫폼 역할

을 동시에 수행하도록 설계되었습니다.

---

## 5. 왜 GitHub를 선택했는가?

GitHub는 단순 코드 저장소가 아니라,
프로젝트 전체 인프라의 핵심 플랫폼 역할을 수행합니다.

프로젝트 내부에서 GitHub는:

- CMS Backend
- Deployment Engine
- Collaboration Platform
- Version Archive
- Authentication Base

역할을 동시에 담당합니다.

---

## 6. GitHub 기반 구조의 장점

### 무료 운영 가능

추가 서버 비용 없이 운영 가능합니다.

---

### 버전 관리

모든 수정 이력이 기록됩니다.

즉:

- 누가
- 언제
- 무엇을

수정했는지 추적 가능합니다.

---

### 복구 가능성

문제가 발생하더라도:

- 이전 커밋 복원
- 히스토리 확인
- 변경 추적

이 가능합니다.

---

### 협업 구조

GitHub는 이미 세계적으로 검증된 협업 플랫폼입니다.

별도의 협업 시스템을 직접 구축할 필요가 없습니다.

---

## 7. 서버를 만들지 않은 이유

프로젝트 설계 당시 가장 중요했던 질문 중 하나는:

> “정말 별도 서버가 필요한가?”

였습니다.

결론은:

> “없어도 충분히 가능하다.”

였습니다.

---

### 서버가 존재할 경우 발생하는 문제

- 운영 비용 증가
- 보안 관리
- 장애 대응
- DB 유지보수
- 운영 책임 증가
- 인수인계 문제

특히 공공기관 환경에서는
이러한 요소들이 장기 운영에 큰 부담이 됩니다.

---

## 8. Markdown 기반으로 설계한 이유

Markdown은 단순 문법이 아니라,
데이터 보존성과 이식성을 위한 전략적 선택이었습니다.

---

### Markdown의 장점

#### 사람이 읽을 수 있는 구조

텍스트 기반이라 가독성이 좋습니다.

---

#### 플랫폼 독립성

어떤 시스템으로도 이전 가능합니다.

---

#### 장기 보존 가능성

최악의 경우 텍스트만 복사해도
데이터 이전이 가능합니다.

---

#### Git과 높은 호환성

변경 사항 비교 및 버전 관리에 최적화되어 있습니다.

---

## 9. React + Docusaurus를 선택한 이유

초기에는 단순 정적 페이지 구조도 고려했습니다.

하지만 최종적으로 React 기반 구조를 선택했습니다.

이유는 다음과 같습니다.

- 동적 위젯
- 컴포넌트 기반 설계
- 반응형 레이아웃
- 실시간 인터랙션
- MDX 지원

등이 필요했기 때문입니다.

---

## 10. MDX 기반 설계

MDX는:

> Markdown + React

구조입니다.

즉:

```mdx
<Column>
<Text>
<Callout>
```

같은 컴포넌트를
Markdown 내부에서 직접 사용할 수 있습니다.

---

### MDX의 핵심 장점

사용자는:

- HTML을 몰라도 되고
- CSS를 몰라도 되며
- React를 몰라도 됩니다.

단순히:

```mdx
<Columns>
```

같은 구조만 복사해 사용해도
복잡한 레이아웃을 구성할 수 있도록 설계했습니다.

---

## 11. CMS 설계 철학

CMS는 사실상:

> “GitHub를 숨기기 위한 사용자 인터페이스”

입니다.

---

## 12. CMS 핵심 목표

### 목표 1

블로그처럼 쉽게 사용할 수 있어야 한다.

---

### 목표 2

코드를 몰라도 수정 가능해야 한다.

---

### 목표 3

실시간 미리보기가 가능해야 한다.

---

### 목표 4

저장 버튼 하나로 자동 배포되어야 한다.

---

## 13. CMS 내부 동작 구조

실제로는 다음과 같은 과정이 내부적으로 수행됩니다.

```txt
저장 버튼 클릭
 ↓
GitHub API 호출
 ↓
Markdown Commit
 ↓
GitHub Actions 실행
 ↓
빌드
 ↓
배포
```

하지만 사용자는 단순히:

> “저장했다”

라고만 느끼도록 설계했습니다.

---

## 14. 중요한 설계 철학

좋은 시스템은
복잡함을 사용자에게 보여주지 않습니다.

복잡한 기술은 시스템 내부로 숨기고,
사용자는 단순함만 경험하도록 만드는 것이 핵심 목표였습니다.

---

## 15. 실시간 미리보기 시스템

CMS에서 가장 중요했던 기능 중 하나는:

> “실제 결과물과 동일한 미리보기”

였습니다.

---

### 기존 Markdown Editor의 문제

일반적인 Markdown Editor는:

- 실제 폰트
- 실제 여백
- 실제 스타일

과 차이가 발생합니다.

즉:
실제 결과물과 편집 화면이 다르게 보이는 문제가 존재합니다.

---

## 16. 해결 방식

실제 `custom.css`를 읽어와:

```css
.theme-doc-markdown
```

를

```css
.toastui-editor-contents
```

로 치환해
에디터 내부에 그대로 주입했습니다.

즉:
실제 홈페이지 스타일을 그대로 미리보기에 적용했습니다.

---

## 17. 결과

관리자는:

> 실제 결과물과 거의 동일한 상태

를 보면서 편집할 수 있게 되었습니다.

---

## 18. 레이아웃 시스템

브로슈어는 종이 기반 디자인 요소가 중요합니다.

그래서:

- A4
- A3
- B5

규격 기반 레이아웃 시스템을 구축했습니다.

---

## 19. 핵심 레이아웃 전략

Docusaurus는 기본적으로 문서 중심 프레임워크이기 때문에
본문 너비가 제한되어 있습니다.

이를 해결하기 위해:

```css
max-width: 100%;
```

를 사용해 제한을 제거하고,
실제 종이 영역만 내부 컨테이너로 다시 제어하는 방식으로 설계했습니다.

## 38. 예상 질문 정리

### Q. 왜 GitHub를 사용했나요?

- 무료 운영 가능
- 협업 기능 제공
- 버전 관리 가능
- 기록 보존 가능
- 자동화 구축 가능
- 이미 검증된 플랫폼

---

### Q. 왜 별도 서버를 사용하지 않았나요?

공공기관 환경에서는:
운영 비용과 유지보수 부담이 매우 크기 때문입니다.

---

### Q. 누가 유지보수 가능한가요?

최소한 Markdown만 이해하면 유지보수가 가능합니다.

또한 최악의 경우:
텍스트 파일만 복사해도 시스템 이전이 가능합니다.

---

### Q. 확장 가능한가요?

새 레포지토리를 생성하기만 하면
자동으로 Hub에 연결됩니다.

---

### Q. 왜 MDX를 사용했나요?

비전공자도 복잡한 UI 컴포넌트를
복사+붙여넣기 방식으로 쉽게 사용할 수 있도록 하기 위함입니다.

## 43. [부록] 실전 컴포넌트 치트시트 (Live Studio용)

라이브 스튜디오에서 복사해서 바로 사용할 수 있는 디자인 컴포넌트 모음입니다.

### 43.1 브로슈어 전용 컬럼 (인쇄 최적화)
가장 안정적인 다단 레이아웃 시스템입니다.
```jsx
<div className="brochure-columns-2">
  <div className="brochure-col">
    내용 1
  </div>
  <div className="brochure-col">
    내용 2
  </div>
</div>
```

### 43.2 고급 타이포그래피 (Text)
지마켓 산스 등 전용 서체를 자유롭게 조절합니다.
```jsx
<Text size="32" weight="800" family="gmarket" color="#0a2463">
  매력적인 제목
</Text>
```

### 43.3 하이라이트 & 강조
```jsx
<Highlight color="#f59e0b">강조 텍스트</Highlight>
```

### 43.4 알림 박스 (Callout)
```jsx
<Callout type="info" title="참고">
  중요한 안내 사항을 입력하세요.
</Callout>
```

## 44. 최종 결론 및 기록

이 시스템의 본질은 기술 과시가 아닌, **사용자의 단순함**을 위해 모든 기술적 복잡함을 시스템 내부(Iframe Renderer, PostMessage, CSS Isolation)로 숨기는 데 있습니다.

- **마지막 업데이트**: 2026-05-16
- **상태**: Live Studio v3 (Stable)
그리고 가장 중요하게 생각했던 부분은:

> 기술적으로는 복잡하지만,
> 사용자는 단순하게 느끼도록 만드는 것.

그것이 잘 설계된 시스템의 핵심이라고 생각합니다.
